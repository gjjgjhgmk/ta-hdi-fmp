# TA-HDI-FMP 研究过程问题与改进总结

## 1. 目标与范围
本轮工作目标是：在不修改 `func/` 底层 FMP 函数的前提下，优化 TA-HDI-FMP 的框架层（RRT + FMP 结合流程），并为后续 ROS2 迁移建立可复现基线。

范围包括：
1. MATLAB 框架层重构（T1~T7）。
2. 测试体系补齐与回归验证。
3. 参数/判定口径调整。
4. Python 核心模块迁移第一阶段。
5. MATLAB 基线导出用于 ROS2 对照。

---

## 2. 研究过程中的主要问题（按时间顺序）

### P1. 主流程参数散落、魔法数过多
问题：初始脚本中的关键参数（RRT、HDI、验收阈值）散落在流程中，不利于实验复现与批量调参。

处理：统一收敛到 `cfg` 配置结构。

---

### P2. `main_core_framework.m` 与 `run_single_trial.m` 双实现分叉
问题：同名逻辑在两个位置独立维护，测试覆盖的版本和主脚本实际运行版本可能不一致，影响可复现性。

处理：
1. 将 `main_core_framework.m` 收敛为入口 + 配置。
2. 统一调用外部 `run_single_trial.m` 与 `run_benchmark.m`。

---

### P3. RRT 代码中的 `pi` 变量冲突
问题：`get_Informed_RRT_Star_path_pool.m` 中局部循环变量曾遮蔽 MATLAB 常量 `pi`，触发运行错误（函数或变量 `pi` 无法识别）。

处理：
1. 循环变量改名（例如 `pidx`）。
2. 角度采样改为安全写法（`builtin('pi')`）。

---

### P4. 测试失败：T3 回归比较对象不一致
现象：`T3` 失败（mean/max shape distance 超阈值）。

根因：测试将 legacy 的“首解”与候选池按代价排序“最优解”直接比较，语义不一致。

处理：
1. 测试改为候选池中匹配比较（不固定 `pool_reg(1)`）。
2. 回归参考优先使用 `old_conv`（收敛路径），`old_first` 仅兜底。

---

### P5. 测试失败：T4 危险/安全样本不足
现象：`T4` 失败（危险区/安全区点数不足，无法比较密度）。

根因：测试场景下 `via_cap` 裁剪导致样本不够，不是 HDI 核心逻辑错误。

处理：测试中单独放大 `via_cap_factor`（仅测试口径调整，不改生产逻辑）。

---

### P6. 路径评分风险项偏保守（易漏检局部高风险）
问题：`score_path.m` 风险项仅用 `mean(ds)`，对局部贴障惩罚不足。

处理：改为混合风险：
`J_risk = 0.7/(min(ds)+0.1) + 0.3/(mean(ds)+0.1)`。

---

### P7. `compare_paths_approx.m` 存在冗余指标
问题：`len_rel` 与 `cost_rel` 在当前实现中语义重复。

处理：去掉冗余 `len_rel` 判定，保留 `cost_rel + 形状误差`。

---

### P8. 候选池去重过粗，影响多样性
问题：仅按成本 + 点数去重，可能误删几何不同路径。

处理：增强去重：成本近似 + 中点几何距离约束。

---

### P9. MATLAB 批量导出执行不稳定（环境问题）
现象：
1. 一次导出超时（>10 分钟）。
2. 后续出现 License Manager -15。

根因：残留 MATLAB `-batch` 进程堆积、许可证服务临时不可达。

处理：
1. 按命令行参数识别并清理残留 `-batch` 进程。
2. 重试导出并确认结果文件更新时间。

---

### P10. Python 迁移时 pip 安装失败（网络代理）
现象：`pip install -e .` 拉取构建依赖失败（代理连接异常）。

处理：
1. 切换为离线路径，不依赖在线安装。
2. 使用 `unittest` + 本地 `PYTHONPATH` 完成测试。

---

## 4. 参数与判定口径调整

根据讨论，已执行以下方向：
1. 曲率阈值：`kappa_pass_thresh` 从 2.0 放宽到 4.0。
2. 碰撞判定口径：引入“硬约束不穿障 + 通过清距”分离思想。

当前观察：即使放宽后，30 次基线导出成功率仍为 0%。
这说明核心瓶颈主要仍在：
1. 穿障（`min_dist < 0` 仍出现）。
2. 轨迹曲率峰值过高（约 7 左右）。

---

## 5. 当前结论

1. 框架层工程质量已显著提升：可配置、可测试、可批量评估、可迁移。
2. MATLAB 与 Python 两侧都建立了可运行的验证通路。
3. 当前性能瓶颈不在“脚本能否跑通”，而在“规划-调制后的可行性质量”（穿障与高曲率）。

---

## 6. 下一步建议（面向 ROS2 与仿真）

1. 保留当前导出基线，新增“宽松判定基线”并行导出（不覆盖原基线）。
2. 在不改 `func/` 的前提下，优先优化框架层约束传递：
   1. 提高候选路径清距权重。
   2. 对 via 点做风险分层（危险区硬约束、非危险区软约束）。
   3. 调制后失败段回退重选候选。
3. 进入 ROS2 第二步：
   1. 建 `ta_hdi_fmp_msgs`。
   2. 建 `ta_hdi_fmp_planner` 最小节点。
   3. 先 RViz 验证，再接 Gazebo。
